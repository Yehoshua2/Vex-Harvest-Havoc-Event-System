Quick start guide

Get a Harvest Havoc competition running in under 10 minutes

This guide walks you through everything from adding teams to running your first match. No setup beyond a browser required.

1

Name your event

Open the app and go to ⚙ Setup. Enter your event name (e.g. "MAD Summer Camp 2026"), choose the number of qualification matches per team, and click Save settings.

Tip: The number of teams × matches per team must be divisible by 4 for a perfectly balanced schedule. For 10 teams, use 4 or 8 matches per team.
2

Add your teams

Still on Setup, enter each team's number and name and click + Add. You need at least 4 teams. Each team gets a unique color dot for quick identification across the app.

Shortcut: Click Load samples to instantly populate 10 sample teams for testing or demos.

Already have a schedule as a CSV file? Paste it into the Import CSV box. Each row must be formatted as:
red1,red2,blue1,blue2

3

Generate the match schedule

Click 🔄 Generate schedule. The system creates a balanced round-robin where teams rotate through different alliance partners. The schedule is created fresh — generating again after adding teams will reset all scores.

Note: Generating a new schedule clears all existing match results. Do this before the competition starts, not during.
4

Call your first match

Go to ☰ Queue. You'll see all pending matches listed in order. Click ▶ next to any match to call it to the field. The match appears in the "On field now" panel. The next match automatically populates "On deck".

Tip: You don't have to go in order. Click ▶ on any match in the queue to jump ahead. Click ↩ to send a match to the back of the queue.
5

Run the match

Go to 📋 Score. Use the scoreboard flow buttons in order:

🎬 Prepare → Live → ▶ Start timer → auto ends, pauses → ▶ Continue to Tele-op → timer ends → ✓ Submit + Finish →

Use the + and − buttons to enter scores for each alliance as the match progresses. Scores update live on the scoreboard window.

✓ After Submit + Finish, the scoreboard automatically shows the winner splash for 5 seconds, then the post-match breakdown.
6

Set up the scoreboard display

Go to 📺 Scoreboard and click Open scoreboard window. Move that window to your projector or second display and make it fullscreen (F11). The preview panel on this page shows a scaled version of what's on the scoreboard at all times.

Tip: Enable Auto-advance (toggle in the Scoreboard tab) so the display transitions automatically between states as you run each match.
7

Save your progress

Click 💾 Save in the top toolbar at any time. This downloads a .json file with all your teams, schedule, and results. Load it back with 📂 Load to resume exactly where you left off.

Note: The app also auto-saves to your browser's local storage after every action, so refreshing the page won't lose your data. The exported file is your backup.

Remote scoring quick setup

To score each alliance on a separate device:

  1. Create a free account at jsonbin.io
  2. Create two bins — one for red, one for blue. Set each to {"type":"score_update","matchId":null,"side":"red","raw":{},"ts":0}
  3. In the app go to 📺 Scoreboard → Remote scoring, paste in both Bin IDs and your Master Key
  4. Click Copy Red scorer link and send it to the red alliance referee's device
  5. Click Copy Blue scorer link and send it to the blue alliance referee's device
  6. Click ▶ Start polling (main) on the main device
Tip: Each scorer only sees their alliance's panel. Scores sync to the main device every 2 seconds automatically.

Event settings

The event name appears on the scoreboard header and in exported file names. The qualification matches per team setting determines how many matches each team plays in the auto-generated schedule. This can be any integer but note that n teams × matches per team must be divisible by 4 for a perfectly balanced schedule — the app will warn you if it isn't achievable.

Team management

Teams are added with a team number (any string — "101", "2996A", "Team 7" are all valid) and a team name. Each team is assigned a unique color dot automatically.

Editing a team

Click the ✏ icon on any team row to edit its number or name inline. Press Enter to save or Escape to cancel. If you change a team number, all references in the match schedule, current match, on-deck match, and sync state are updated automatically.

Removing a team

Click 🗑 and confirm. This removes the team but does not remove their completed match results — historical data is preserved.

Schedule generation

Clicking 🔄 Generate schedule builds a balanced round-robin where teams are paired through a greedy algorithm that always picks the team with the fewest matches so far. The algorithm shuffles candidates before picking to vary alliance partners across matches.

Important: Generating a new schedule resets all match results and clears the current match state. Always generate before competition starts.

CSV import

You can import a pre-built bracket by pasting CSV into the Setup page. Each data row must be exactly: red_team1, red_team2, blue_team1, blue_team2. Lines starting with # are treated as comments and ignored. Imported matches are appended to any existing schedule rather than replacing it.

Calling matches to the field

The Queue page shows all pending matches. The first pending match is highlighted as "Next". Click ▶ on any match row to call it to the field immediately — you are not constrained to go in order. When a match is called, the next pending match automatically moves to "On deck".

Skip & reorder

Click ↩ on any queued match to move it to the end of the pending list. This is useful when a team is not ready. The ↩ button also appears on the "On field now" card to skip the current match mid-flow.

Extra matches

Click + Extra in the Queue header to manually create a match with any four teams. Extra matches are tagged differently from qualification matches in rankings — the Overall rankings tab includes them, the Qual only tab excludes them.

Completing matches

Click ✓ Complete to mark the current field match as done and advance the queue. If no score has been entered, you'll be prompted to confirm completing with 0-0. The preferred way to complete a match during normal operation is via Submit + Finish → on the Score page, which also triggers the scoreboard flow.

Match flow (Score page)

The Score page has a flow bar with three buttons that control both the timer and the scoreboard display state:

🎬 Prepare → LiveSCOREBOARD

Switches the scoreboard to the Live state and locks the live score feed so the display shows 0-0 while you prepare the field. Press this when teams are set up and you're about to start.

▶ Start timerTIMER

Unlocks the live score feed and starts the match timer counting down from 3:00. The timer automatically pauses at 2:40 (when the 20-second autonomous period ends) and displays "PAUSED". Scores entered before and after this point are treated the same — auto vs tele-op is tracked by which score fields you fill in, not by when you tap.

▶ Continue to Tele-opTIMER

Resumes the timer from wherever it paused (typically 2:40) and unlocks the live score feed. Press this after scoring the auto period. The timer will then count down through tele-op and end game, pausing the score feed again when it reaches zero.

✓ Submit + Finish →SCORE

Saves the current scores, marks the match as complete, advances the queue, and — if auto-advance is on — transitions the scoreboard through Splash (5 seconds) then Post-match. This is the primary way to end a match.

✓ Submit scoresSCORE

Saves the current scores without completing the match or advancing the scoreboard. Use this to save mid-match progress or to re-enter scores after reviewing, without triggering the scoreboard transition.

Score entry

The Score page shows two panels side by side — one for each alliance. Use + and − to increment/decrement each scoring element. Fields that have a maximum (LEAVE, climbs) will not go above their limit. The total score for each alliance updates in real time at the bottom of the page and on the scoreboard window.

Scores are broken into sections: Auto, Tele-op fruit, Crates, Hives, End game climbs, and Penalties.

Hive multipliers

Each alliance has two hive checkboxes — Berry hive and Apple hive. When a hive ends the match in the alliance's alliance zone, check the corresponding box. This changes the per-crate point value automatically:

  • Berry hive in zone: Berry crates change from 18 pts → 27 pts each
  • Apple hive in zone: Apple crates change from 24 pts → 36 pts each

The hint text under each crate row updates to reflect the current effective point value, and a ✓HIVE badge appears next to the crate label when active.

Penalties & disqualification

Penalties are entered as the number of fouls received by that alliance (i.e. fouls committed by the opposing alliance). Minor fouls add 5 pts, major fouls add 10 pts to the receiving alliance. A red card cancels the entire alliance's score to zero — checking this box overrides all other scoring for that alliance.

Editing scores after submission

Go to 🕐 Results. Every completed match has an ✏ button on the right. Clicking it opens a full score edit modal with the same scoring form pre-populated with the saved values. Save changes and rankings update immediately.

Scoreboard display states

The scoreboard has four states, controlled either manually from the Scoreboard tab or automatically via auto-advance:

StateWhat it shows
📋 Pre-matchThe upcoming match's teams (red and blue alliance), plus the on-deck match below. "PLEASE BRING YOUR ROBOT TO THE FIELD" prompt.
▶ LiveBoth alliance scores updating in real time, per-category stat breakdown (berries, apples, crates, climb pts), the match timer, and a live rankings sidebar.
🏆 SplashFull-screen winner announcement with the winning alliance name in large text, final score, and team numbers. Color-coded background (red/blue/grey for tie).
📊 Post-matchFinal scores for both alliances with a full category-by-category breakdown and updated rankings list.

Timer controls

The timer can be controlled manually from the Scoreboard tab regardless of auto-advance:

  • ▶ Start — begins counting down
  • ⏸ Pause — freezes the timer without locking scores
  • ↺ Reset — returns to the selected duration
  • Mode selector — Full match (3:00), Auto only (0:20), Tele only (2:40), End game (0:30)
  • Show timer toggle — hides/shows the timer on the scoreboard without stopping it

When the timer expires, the live score feed is automatically locked (SCORES LOCKED badge appears on scoreboard).

Auto-advance

When the Auto-advance toggle is on, the following transitions happen automatically:

Call match to field → Scoreboard goes to Pre-match, timer resets
🎬 Prepare → Live → Scoreboard goes to Live, scores locked
▶ Start timer → Timer starts, scores unlocked
Auto ends (2:40) → Timer pauses, scores locked
▶ Continue to Tele-op → Timer resumes, scores unlocked
Timer reaches 0:00 → Scores locked
Submit + Finish → → Splash (5s) → Post-match

Hide live scores

The Hide live scores toggle in the Scoreboard tab replaces the live score numbers on the display with — dashes. The scores are still being tracked and calculated — this is purely cosmetic for situations where you don't want the audience to see the running total mid-match.

How ranking works

Teams are ranked by wins first, then total accumulated points, then opponent points (as a tiebreaker), then highest single-match score. The Overall tab includes all match types. The Qual only tab excludes extra matches.

Each team accumulates the alliance score from every match they play in. There are no score drops in this system.

Tiebreakers

When two teams have the same number of wins, the tiebreaker order is:

  1. Higher total points accumulated across all matches
  2. Lower opponent points (teams that won against higher-scoring opponents rank higher)
  3. Higher single-match high score

Remote scoring setup

Remote scoring lets one referee per alliance score on their own phone or tablet. It requires a free account at jsonbin.io.

  1. Create two bins at JSONBin — one for red, one for blue
  2. Set each bin's initial content to {"type":"score_update","matchId":null,"side":"red","raw":{},"ts":0} (change "side" to "blue" for the blue bin)
  3. In the app, go to Scoreboard → Remote scoring and paste both Bin IDs and your Master Key
  4. Click Copy Red scorer link — this generates a URL with the credentials embedded. Send to the red scorer
  5. Click Copy Blue scorer link — send to the blue scorer
  6. Click ▶ Start polling on the main device

Sync behavior

When a scorer opens their link, they see only their alliance's scoring panel. Every time they tap + or −, their scores are pushed to their bin. The main device polls both bins every 2 seconds and merges the incoming scores into its own form, updating the live totals and the scoreboard display.

Two separate bins are used so red and blue can push simultaneously without overwriting each other's data.

Limitation: The free JSONBin tier allows 10,000 requests per month. At a 2-second poll rate on the main device (pulling from 2 bins), you use ~86,400 requests per day of competition. Consider increasing the poll interval for longer events.

Save & load

Click 💾 Save to download a .json file containing all state: event settings, teams, match schedule, results, and dark mode preference. Click 📂 Load to restore from a saved file. The app also writes to browser local storage after every change as a safety net.

Clearing data

The 🗑 Clear button in the header opens a menu with three options:

  • Clear all scores — resets every match to "pending" and removes all results. Teams and schedule are preserved.
  • Clear schedule + scores — removes all matches and results. Teams are preserved.
  • Clear everything — full reset. Only event settings are preserved.
Technical reference

Harvest Havoc Event System — Developer Documentation

Architecture overview, data model, function reference, and extension guide for anyone modifying the codebase.

Architecture overview

The entire application is a single self-contained HTML file with no build step, no framework, and no server. All logic runs in the browser. The file is structured as:

HTML structure

Semantic page divs controlled by visibility. Each "tab" is a .page div shown/hidden by the P() router.

CSS custom properties

All colors and spacing use CSS variables in :root and [data-dark="1"] overrides. Dark mode is toggled by setting data-dark on #app.

State object S

A single global S object holds all app state. save() serializes it to localStorage. Every mutation calls save().

Scoreboard renderer

buildSBHTML(payload) is a pure function: payload in, HTML string out. Used both for the inline preview and the separate window.

State object

The global S object is the single source of truth. Its shape:

KeyTypeDescription
S.eventobjectname, twPer (matches per team)
S.teamsTeam[]Array of {num, name, color}
S.matchesMatch[]All matches — see Match shape below
S.resultsResult[]Completed match scores — see Result shape below
S.currentMatchMatch|nullMatch currently on the field
S.onDeckMatchMatch|nullNext match in queue
S.darkModebool|nullnull = follow OS preference

Match shape

{
  id:     string,        // 'Q1', 'E3', etc.
  num:    number,        // display number
  type:   'qual'|'extra',
  red:    string[],      // [teamNum, teamNum]
  blue:   string[],      // [teamNum, teamNum]
  status: 'pending'|'done'
}

Result shape

{
  matchId:   string,     // references Match.id
  redScore:  number,
  blueScore: number,
  raw: {
    red:  RawScore,
    blue: RawScore
  },
  ts: number             // Date.now() timestamp
}

RawScore shape

{
  auto_leave:  number,   // 0-2
  auto_berry:  number,
  auto_apple:  number,
  tele_berry:  number,
  tele_apple:  number,
  cr_mixed:    number,
  cr_berry:    number,
  cr_apple:    number,
  hive_berry:  boolean,
  hive_apple:  boolean,
  climb_lo:    number,   // 0-2
  climb_hi:    number,   // 0-2
  minor_foul:  number,
  major_foul:  number,
  redcard:     boolean
}

Scoring formula

Implemented in calcD(ctx, side). The formula is:

score =
  auto_leave  × 5  +
  auto_berry  × 12 +
  auto_apple  × 20 +
  tele_berry  × 3  +
  tele_apple  × 5  +
  cr_mixed    × 12 +
  cr_berry    × (hive_berry ? 27 : 18) +
  cr_apple    × (hive_apple ? 36 : 24) +
  climb_lo    × 30 +
  climb_hi    × 50 +
  minor_foul  × 5  +
  major_foul  × 10

if (redcard) score = 0

ctx is either 'main' (live score page) or 'edit' (edit modal). This namespaces the DOM element IDs so both forms can exist simultaneously without collision. Element IDs follow the pattern ${ctx}_${side}_${key}.

Scoreboard renderer

buildSBHTML(payload) is a pure function that takes a payload object and returns a complete HTML string. It is used in two places:

  • updSBPrev() — sets innerHTML of the preview div on the Scoreboard control page
  • The scoreboard window's receivePayload(p) — sets innerHTML of the root div in the popup window

Because the function is serialized into the popup window via buildSBWinHTML() using .toString(), it must be a pure function with no references to outer scope variables except SC() (the stat card helper). Any utility functions it calls must also be serialized or inlined.

Payload shape

{
  state:      'pre'|'live'|'splash'|'post',
  ev:         string,           // event name
  cur:        MatchView|null,   // current match with team objects
  nxt:        MatchView|null,   // on-deck match
  nxt2:       MatchView|null,   // match after on-deck
  result:     ResultView|null,  // current match result
  lastRes:    ResultView|null,  // last completed result (for splash/post)
  lR:         number,           // live red score (from DOM)
  lB:         number,           // live blue score (from DOM)
  lRaw:       {red, blue},      // live raw scores
  livePaused: boolean,
  hideScores: boolean,
  tmr: {
    v:    number,               // seconds remaining
    mx:   number,               // timer max
    run:  boolean,
    en:   boolean,              // show timer on display
    fmt:  string,               // '2:40'
    per:  string                // 'AUTONOMOUS' | 'TELE-OPERATED' | 'END GAME'
  },
  ranks: TeamStat[]
}

Remote sync system

The sync system uses two JSONBin bins (one per alliance) as a cloud clipboard. The scorer device writes to its bin; the main device polls both bins.

SCORER DEVICE (red) JSONBIN (red bin) MAIN DEVICE ───────────────────────────────────────────────────────────────────────────── adj() called └─ syncPush({side:'red', raw}) ──PUT──▶ bin overwrites polls every 2s └─ syncPull(redBin) └─ syncPull(blueBin) └─ merge into DOM └─ recalc('main') └─ pushSBLive()

Key design decisions:

  • Two bins prevent race conditions — red and blue never write to the same bin
  • syncPush does a direct PUT (no read-modify-write), so it's a single atomic operation
  • The main device uses Promise.all to pull both bins in parallel each poll cycle
  • The scorer device detects its role from URL params set by copyScorerLink()
  • Scorer role is detected in an IIFE at script start, before initUI(), so it's set before any rendering occurs

Timer implementation

The timer uses setInterval with a 1-second tick. On each tick it decrements tmrVal and calls pushSBTmr() to update the scoreboard. Auto-pause logic:

// Inside tmrStart() setInterval callback:
if (tmrVal === 160 && tmrMax === 180) {
  // Auto period just ended — pause
  clearInterval(tmrInt);
  tmrRun = false;
  pushSBTmr();
  return;
}

The tmrMax === 180 guard ensures auto-pause only fires in full-match mode. When running auto-only mode (tmrMax = 20) it counts straight to zero.

Known limitation: setInterval is not guaranteed to fire at exactly 1000ms in all browser conditions (tab backgrounding, heavy CPU load). For competition-critical timing, consider using Date.now() deltas instead of a tick counter.

Page router

function P(id) {
  // Hide all pages
  document.querySelectorAll('.page').forEach(p => p.classList.remove('active'));
  // Show target
  document.getElementById('page-' + id).classList.add('active');
  // Update nav button active state
  document.querySelectorAll('.nav-btn').forEach(b => b.classList.remove('active'));
  [...document.querySelectorAll('.nav-btn')]
    .find(b => b.getAttribute('onclick') === `P('${id}')`)
    ?.classList.add('active');
  // Page-specific render
  if (id === 'queue') renderQueue();
  if (id === 'score') renderScore();
  // ...etc
}

All render functions are called on navigation, not on data change, except for functions that push to the scoreboard (pushSB(), pushSBLive()), which are called from mutation sites.

Dark mode

Dark mode is implemented entirely with CSS custom property overrides. The [data-dark="1"] selector on #app redefines all :root variables. Toggling dark mode only requires setting that attribute — no class changes, no style recalculation beyond what the browser does naturally.

// Three-state logic:
// S.darkMode === null  → follow OS (matchMedia listener)
// S.darkMode === true  → always dark
// S.darkMode === false → always light

function initDark() {
  S.darkMode === null
    ? applyDark(window.matchMedia('(prefers-color-scheme:dark)').matches)
    : applyDark(S.darkMode);
}

Adding a new scoring element

To add a new scoring field (e.g. a new game piece type), you need to touch four places:

  1. RawScore shape — add the new key (e.g. new_piece: number)
  2. bPanel() — add a .si row with the counter and adj() call
  3. calcD() — add the scoring formula line
  4. buildSBHTML() — add it to the live breakdown or post-match row function if you want it visible on the scoreboard

The doSubmit() and saveScores() functions call calcD() which reads from DOM, so they don't need changing — they'll pick up the new field automatically.

Extending the sync system

To change the sync backend from JSONBin to something else (e.g. Firebase Realtime Database, Supabase, or a custom WebSocket server), you only need to replace two functions:

async function syncPush(payload) {
  // Write payload to your backend for payload.side ('red' or 'blue')
}

async function syncPull(binId) {
  // Read and return the stored record for the given channel ID
  // Return null on failure
}

Everything else — the poll loop, DOM merging, scorer role detection — is backend-agnostic and doesn't need to change.

Local storage

The app uses localStorage key 'hhf'. On load it falls back to older keys 'hh_v3', 'hh_v2', 'hh_v1' for migration. The stored value is JSON.stringify(S). There is no schema versioning — if you add new fields to S, add defaults in the load() function's fallback logic.

Known issues & limitations

  • Timer drift: setInterval can drift under load. Use Date.now() deltas for production precision.
  • JSONBin rate limits: Free tier is 10k requests/month. At 2s poll + 2 bins = ~86k/day. Increase JSONBIN.pollMs for longer events.
  • Popup blockers: The scoreboard window uses window.open(). Some browsers block this unless triggered directly by a user click. The Open button is a click handler, so it should work — but browser extensions may interfere.
  • Scoreboard window reload: If the scoreboard window is reloaded, it loses its receivePayload reference. Close and re-open from the main device.
  • Single localStorage key: Multiple events in the same browser share state. Use the Save/Load file system for managing multiple events.
